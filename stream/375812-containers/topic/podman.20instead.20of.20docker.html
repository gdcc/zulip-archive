Philip Durbin 🚀 (Jul 10 2026 at 20:41):

"1777 reminds me of 777 which I've always heard is poor security" -- me

"Fair instinct — 777/1777 on a real filesystem is usually a red flag because it lets any user read/write/delete anyone else's files. Here the risk profile is different:

  • It's a private, ephemeral tmpfs inside one container's mount namespace — not shared with the host or other containers, and wiped on container stop. There's no "other users" to protect against; only the one process (payara) writes there anyway.
  • The 1 is the sticky bit — same as the real /tmp on every Linux system. It means only a file's owner (or root) can delete/rename it, even though the directory is world-writable. So one process can't clobber another's temp files even in a multi-tenant /tmp.
  • The alternative (uid=/gid= mount options) would be tighter — restricting write access to just uid 1000 — but Podman's Docker-API emulation rejects that option, which is the whole reason we're here.

So mode=1777 here is the same trust model as your OS's real /tmp, not an open-permissions container image or host mount. If you wanted tighter isolation you could restrict it to mode=1770 and have the container's entrypoint chown /tmp to the payara uid at startup — but that requires editing the image/entrypoint, not just the compose file. For this demo compose setup, 1777 is a reasonable, standard tradeoff.

✻ Sautéed for 9s" -- Claude
